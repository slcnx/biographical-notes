项目经验

运维平台开发

项目背景
项目中有很多脚本，但是不是每个运维都有能力去使用他，还有一些重复性低效的操作需要解决。
项目实现

将应用效率脚本，通过 web 界面输出，运维不再需要关心脚本的环境问题，有一个统一的界面可以使用。将一些低效重复操作，通过界面一键完成信息获取、操作执行。

项目业绩
  • 1、原始处理一个重复问题，需要3分钟，降低到1分钟。
  • 2、将30分钟处理，或需要定时处理的任务，通过平台来处理，解放运维人员的生产力。

Spring Cloud链路追踪系统完善

项目背景
公司核心业务项目出现用户反馈前端操作慢，需要进行对业务分析。
项目实现

通过对网页操作，从前端、SLB 、后端逐一定位，前端页面加载没有问题，入口流量正常，接口响应慢。但是没有使用到 Spring Cloud的 Slueth+Zipkin ，引入此平台，可定位到我们公司产品在调用第3方接口、mysql 查询语句有慢 SQL 、登陆逻辑有一个循环取值的操作，通知相应人员，优化慢 SQL ，开发优化逻辑。

项目业绩
  • 1、之前处理问题，需要依赖开发的直觉看代码是哪里慢，到从监控平台可以快速定位问题，定位问题提升效率。
  • 2、在优化后，网页核心接口加载速度从3s优化到1s左右。

优化CICD流程

项目背景
我们公司多个项目，发布过程均可以使用，但是测试人员反馈我们的发布非常缓慢，进行优化发布过程。
项目实现

对发布每个步骤进行优化，拉代码、构建、发布优化。拉代码使用 https/ssh 加密传输，构建过程，从国外的镜像站点，迁移到国内的镜像站点，同时镜像使用多阶段构建提升了构建速度，和开发人员沟通，优化了前端代码体积，使用 tree shaking 、动态导入、gzip 压缩等机制，优化 pod 模板，引入 startup 勾子，可以快速让服务在线。

项目业绩
  • 1、发布过程，从原始5-6min 缩减到1-3min ，提升发布效率。
  • 2、从原始的http拉代码配置，修改为ssh,提升了代码传输的安全性。
  • 3、优化了pod模板，让Java服务启动就绪从1分钟缩减到20-30s。

商城项目基于灰度发布

项目背景
公司某个新上线的商城项目，需要稳定的发布，原始使用 Kubernetes 的滚动更新有一些问题：发布过程中流量正常与否无法检验，发布稳定性不能提升。
项目实现

结合 Kubernetes 生态，调研后采用 flagger+ingress+prometheus 发布，使用基于权重的发布方式，当发布流量验证出现问题，可以自动回滚发布。

项目业绩
  • 1、引入灰度发布组件，提升发布稳定性，不再出现发布之后，业务不可用的问题。
  • 2、提升服务稳定性，给商城介绍添加一个新的卖点，采用灰度发布方式，提升用户使用体验。

日志统一管控

项目背景
项目写入日志前期由开发在代码中写入数据库，也尝试过插件写入到外部系统中，由于不同开发人员均需要引入此模块，同时没有写入到存储中的日志存在一些问题：查询难，不易搜索等问题。
项目实现

对pod模板进行改造，方便收集容器的 stdout、stderr 日志。业务不再需要写入日志，直接通过统一展示平台查看日志，和开发人员沟通通过正则可获取到核心业务的报错，并通知到群。

项目业绩
  • 1、解决开发人员的代码冗余问题，避免部分业务未采集到日志的问题。
  • 2、解决开发人员的查看日志问题，提升了日志排错的效率，从原来的5-10分钟定位问题，缩短到1-3分钟定位问题。
  • 3、将核心业务日志监控起来，提升了业务问题定位能力，从原来用户反馈问题，到自主解决问题。

项目迁移及容器化

项目背景
入职后接手微知秀是一个 docker-compose 单机多服务项目，语言是 Java ，面临的问题服务不是高可用，应用无弹性、配置冗余。
项目实现

为解决以上问题，调研当时容器集群解决方案，最终选择 Kubernetes ，首先在本地规划 pod、service 网络，选型CNCF认证过的k8s部署工具 Kubeasz 搭建二进制k8s集群，同时将业务数据同步到本地 ，定义 pod 模板 ，确保服务高可用、应用弹性、配置中心统一管理服务配置，而后迁移服务到云上，后灰度策略发布新集群，输出项目架构图。

项目业绩
  • 1、迁移成功后，单体到多机，提升服务容错。
  • 2、运维操作规范化，不存在大量冗余的配置。
  • 3、应用可弹性，当负载高时，可自动伸缩副本，实现高峰流量处理。
